The Dojo Framework

Dojo

  • Core Framework Components
  • Anything that begins dojo.*
  • Explicitly excluded is anything concerning rendering on to the page

Dijit

  • Dojo's Widget Framework
  • "Widget" doesn't just mean small components like buttons or text fields
  • The widget framework allows for composability: you can nest components inside of each other to build up a more powerful widget
  • Anything that begins dijit.*

Dojox

  • DojoX is an area for development of extensions to the Dojo toolkit. It is a repository for more stable and mature extensions and also acts as an incubator for experimental code, a testbed for additions to the main toolkit. Unlike Dojo and Dijit, DojoX is managed by subprojects, each of which has at least one module, a sponsor and a mission statement.

Asynchronous Module Definition

Problem:

How do you organize your javascript project so that you minimize code duplication and maximize reuse of successful patterns?

Answer?

  • With RequireJS!
  • It's framework-independent so you can mix-and match modules
  • It can be easily optimized using an r.js compiler*
  • Provides a very flexible module definition syntax that allows you to easily express Java-style patterns (and enhance them with some javascript additions)

AMD Example Class

define("Printer", [
    "dojo/_base/declare"
    ], function(declare) {
    return declare("Printer", null, {
        print: function(input) {
            dojo.create('div', {
                innerHTML: input
            }, dojo.body());
        }
    });
});
require([
    "dojo/ready",
    "Printer"
    ], function(ready, Printer) {
    ready(function() {
        var printer = new Printer();
        printer.print("hello1234");
    });
});​

How about some mixins?

Mixins

  • Mixins allow you to break functionality into logical groupings
  • You can then "mixin" desired capabilities for your class
  • Can be a great tool for fighting cyclomatic complexity

Operation.js:

define("Operation", ["dojo/_base/declare"],function(declare) {
    return declare("Operation", null, {   
        base: null
    });
});

AddMixin.js:

define("AddMixin", ["dojo/_base/declare", "Operation"], function(declare,Operation) {
    return declare("AddMixin", [Operation], {
        add: function(x, y) {}
    });
});

CalculatorImpl.js:

define("CalculatorImpl", ["dojo/_base/declare", "AddMixin"],
function(declare,AddMixin) {
    return declare("JavascriptMixin", [Programmer, AddMixin], {
        
    });
});

init.js:

require(["dojo/ready","CalculatorImpl"], function(ready, CalculatorImpl) {
    ready(function() {
        var calculator = new CalculatorImpl();
        calculator.add(1,1);
    });
});​​​

Keepin' it D.R.Y.

AMD allows you to better adhere to the design approach "Don't Repeat Yourself ." Use AMD to build up units of functionality and then mix them together to create your desired class.

Connect, Deferred, and Hitch, Oh My!

Connect

  • A really cool function that allows you to chain methods together dynamically
  • It is preferred that you use connect to a more traditional pub/sub approach.
    • When you use quotes for method names, the methods will not be resolved until execution time.
    • That means you can wire objects together before they've finished initializing (even before the methods you're wiring exist on the prototype chain)
              var guy1 = {
                  speak: function(){console.log("hello");}
              };
              var guy2 = {
                  speak: function(){console.log("there");}
              };

              dojo.connect(guy1,'speak',guy2,'speak');
              guy1.speak();
              
              > hello
              > there

Dojo Deferred

  • Dojo has the ability to manage communication between asynchronous callbacks
  • You can chain multiple callbacks together so that individual asynchronous steps happen in order
  • Deferreds also have success and error callbacks so you can write recovery handling into your application
    var deferred = new dojo.Deferred();
    setTimeout(function() {
        echo("Timeout firing");
        deferred.resolve({
            called: true
        });
    }, 1000);

    echo("Initial page content.");

    deferred.then(function() {
        echo("Deferred firing");
    });
      
              > Initial page content
              > Timeout firing
              > Deferred firing

Dojo Layers

Dojo Layers

  • Each dependency you bring in results in a HTTP call to the server.
  • Consequently having a lot of dependencies can significantly impact performance.
  • Fortunately Dojo provides a mechanism through which you can aggregate dependencies into a single optimized file
  • If you want, you can get clever and write a resource loader for your widgets. You can load individual files when debugging your widget and pull in the layer when debug mode is turned off.
    dependencies ={
      stripConsole: "normal",
      selectorEngine: "acme",
      layers:  [
        {
          name: "../synapse/widgetlayers/example.js",
          dependencies: [
            "dijit.layout.BorderContainer",
            "dijit.layout.TabContainer",
            "dijit.layout.ContentPane",
            "dojox.data.FlickrRestStore",
            "dojox.image.LightboxNano",
            "synapse.common.applicationController.SynapseController",
            'synapse.components.synapseControls.ControlBar',
            'synapse.common.DataController'
          ]
        }
      ]
    }

Templates

Templates

  • In Dojo you can define templates outside your javascript to express the look and feel of your application
  • Dojo templates just do string substitution and do not handle a lot of the advanced features you find in something like Ext's XTemplates (conditional processing, basic math function support, inline code execution against template variables, and built-in operators like iterators.
  • Using templates means that you don’t have to mix HTML markup inside JavaScript logic. It will make your code cleaner and allow easier modifications to look and feel.

Dojo Template Example

   
    <div data-dojo-type="example.SomeWidget">
      <p>This is arbitrary content!</p>
      <div data-dojo-type="dijit.form.Button">${buttonLabel}</div>
      <p>More arbitrary content!</p>
    </div>

Internationalization

Internationalization

  • Dojo supports internationalization through mixins
  • You can either mix your labels into your class (which makes it easier to use them in templates) or put them into some object for later use

Internationalization Example

./nls/exampleWidget.js:

 define({ root:
    ({
      label1: "labelOne"
    })
  })

./ExampleWidget.js:

define("example/exampleWidget",
    [
        "dojo/_base/declare",
        "dojo/i18n!./nls/myLocalization"
    ],
    function(declare, myLocalization){
      return declare("example.exampleWidget", null,{
        constructor: function(){
          dojo.safeMixin(this, 
            dojo.i18n.getLocalization("exampleWidget", "myLocalization"));

            //go use this.label1 somewhere
        }
      } 
    }
  );

Integrating Dojo with Maven

The Dojo Buildtool

  • Dojo provides an implementation of the RequireJS compiler that is used to build the Dojo project
  • The actual dependency resolution and layer building occurs inside an instance of node.js they setup as part of the compile
  • There is currently no maven plugin for dojo, so you will need to use the exec-maven-plugin to invoke the dojo build chain
  • Furthermore, the compile process needs access to the whole Dojo project. Therefore whenever you want to compile you need to make sure the whole dojo library has been expanded to an accessible location

Example POM

            <plugin>
                <groupId>org.codehaus.mojo</groupId>
                <artifactId>exec-maven-plugin</artifactId>
                <version>${exec.maven.plugin.version}</version>
                <configuration>
                    <executable>${java.home}/bin/java</executable>
                    <workingDirectory>
                        ${dojo.source.local.unpacked.location}/util/buildscripts
                    </workingDirectory>
                    <arguments>
                        <argument>-Xms${dojo.build.xms}</argument>
                        <argument>-Xmx${dojo.build.xmx}</argument>
                        <argument>-classpath</argument>
                        <argument>${dojo.build.classpath}</argument>
                        <argument>org.mozilla.javascript.tools.shell.Main</argument>
                        <argument>./../../dojo/dojo.js</argument>
                        <argument>baseUrl=./../../dojo</argument>
                        <argument>load=build</argument>
                        <argument>copyTests=${dojo.build.copyTests}</argument>
                        <argument>profileFile=${dojo.build.profileFile}</argument>
                        <argument>action=${dojo.build.action}</argument>
                        <argument>version=${dojo.build.version}</argument>
                        <argument>releaseName=${dojo.build.release.name}</argument>
                        <argument>releaseDir=${dojo.build.release.location}</argument>
                        <argument>layerOptimize=${dojo.build.layerOptimize}</argument>
                        <argument>internStrings=${dojo.build.internStrings}</argument>
                        <argument>optimize=${dojo.build.optimize}</argument>
                        <argument>cssOptimize=${dojo.build.cssOptimize}</argument>
                    </arguments>
                </configuration>
                <executions>
                        <execution>
                        <id>dojo-custom-build</id>
                        <phase>compile</phase>
                        <goals>
                            <goal>exec</goal>
                        </goals>
                    </execution>
                </executions>
            </plugin>

Making TDD work with Dojo

Test Driven Development

  • Test driven development is officially defined (on Wikipedia) as follows: "In test-driven development a developer creates automated unit tests that define code requirements then immediately writes the code itself."
  • In practice I have had trouble following this paradigm. I prefer to write and test incrementally.
  • I like to write one basic test case that covers the expected use case for my code
  • I'll then add tests as bugs / features are reported

What makes TDD hard in Javascript?

  • Testing algorithms and datastructures is actually pretty straight forward.
  • You start to run into trouble when you want to test UI elements, and that's not a problem that's unique to Javascript.
  • Most test harnesses for javascript run inside a headless browser (like Rhino).
  • Consequently you end up with a divided project: you have to arbitrarily separate your UI code from the rest of your project so you can test.

So what to do...

The New Hotness: JSTestDriver

JSTestDriver

  • JSTestDriver allows you to slave different browers (even from remote hosts like a mobile phone) to a locally run test server
  • You can then incrementally test the exact code that would be run in the browser
  • Has a major advantage over things like the Dojo Objective Harness or Selenium because of its speed.
  • Running a DOH test suite can take minutes because it recreates each 'click' you make with your mouse and plays them back
  • JSTestDriver does not record anything: you have to code everything (including mouse clicks) that you want to happen

Basically you can write this:

    queue.call('Load the component on the page',function(callbacks){
        buttonGroup = new synapse.components.buttonGroup.ButtonGroup();
        button1 = new synapse.components.buttonGroup.SelectableButton({
            iconClass: "synapseSearchIcon",
            showLabel: false,
            buttonGroup: buttonGroup
        }).placeAt(dojo.byId('myDiv'));
        button2 = new synapse.components.buttonGroup.SelectableButton({
            iconClass: "synapseSearchIcon",
            showLabel: false,
            buttonGroup: buttonGroup
        }).placeAt(dojo.byId('myDiv'));

    });

    queue.call('Execute a test 1',function(callbacks){
        button1._onClick({
            preventDefault:function(){},
            stopPropagation:function(){}
        });
        assertTrue(button1.checked);

        button2._onClick({
            preventDefault:function(){},
            stopPropagation:function(){}
        });
        assertTrue(button2.checked);
        assertFalse(button1.checked);

    });

So testing is pretty much solved, right?

The Fine Print

  • Test suite needs explicit declarations for anything that is asynchronous, which can mean reaching into your library code to wrap things correctly (like deferred)
  • Since there is no HTML at all, you have to inject you library onto the page. For things like Google Closure, this can mean needing to rewrite their supplied loader
  • Sometimes your test suite can just 'break' and you need restart your IDE, server, browser, etc until it starts working again
  • Anytime you change code, you need to manually flush your browser cache. If you don't you run the risk of testing cached code vs. the changes you've made.
  • Since many libraries are dynamically loaded, you need to proxy your javascript through the test framework to avoid XSS errors.
